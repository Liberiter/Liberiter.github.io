> **이 장의 질문** — 정규식 한 줄은 어떤 절차로 "찾았다/못 찾았다"를 판정하는가?

**이 장을 마치면 할 수 있는 것**

- uv로 Python 3.14 실습 환경을 설치하고 검증 명령 세 개를 모두 통과시킨다
- REPL에서 re.search로 첫 리터럴 매칭을 실행하고 매치 위치를 읽어낸다
- raw string 유무에 따라 같은 패턴의 결과가 갈라지는 사례를 재현한다
- 매치 실패(None)를 분기 처리해 NoneType 오류 없이 결과를 출력한다
- UTF-8 실행 규약(-X utf8)을 적용해 양 OS에서 동일한 실행 결과를 확인한다

청람항의 서기 담로는 아침마다 입출항 명부를 손가락으로 짚어 내려간다. 은파호가 들어왔는지, 첫서리호 점검이 잡혀 있는지. 이 장에서 하는 일이 정확히 그것이다 — 손가락 대신 정규식 한 줄을 쓴다는 점만 다르다.

정규식은 도구이기 전에 작은 언어다. 문자열 하나를 놓고 "찾았다" 또는 "못 찾았다"라는 판정을 내리는 언어. 그 판정이 어떤 절차로 내려지는지를 처음부터 끝까지 따라가는 것이 이 장의 등뼈다. 재료는 명부 한 장, 함수 하나(`re.search`), 그리고 판정 결과를 담는 객체 하나면 충분하다.

**시작 전 점검.** 터미널을 열고 명령 한 줄을 실행해 본 적이 있는가? 디렉터리를 옮기고(`cd`), 실행 결과를 눈으로 읽는 정도면 충분하다.

<details>
<summary>답 확인</summary>

경험이 없다면 OS 기본 터미널(macOS는 터미널 앱, Windows는 Windows Terminal)을 열어 `cd`와 `ls`를 몇 번 실행해 보고 시작하는 편이 좋다. 이 장의 설치 절차가 전부 터미널에서 진행된다.

</details>

**시작 전 점검.** 파이썬 기초 세 가지 — 문자열 리터럴(`"..."`), 함수 호출(`f(x)`), `if` 분기 — 를 읽고 쓸 수 있는가? 예제에 나오는 `for` 루프·리스트·f-string은 읽을 수 있는가?

<details>
<summary>답 확인</summary>

쓸 줄 알아야 하는 것은 앞의 셋이고, `for` 루프·리스트·f-string은 예제를 따라 치며 읽어낼 수준이면 충분하다. 함수 정의(`def`)는 이 장 마지막 문제에서 처음 나오는데, 모범답안의 형태를 그대로 따라도 된다. 클래스나 모듈 작성 경험은 필요 없다. `if`가 낯설다면 파이썬 공식 튜토리얼의 제어 흐름 절만 훑고 돌아오라.

</details>

## 1.1 작업대 차리기 — uv, Python, ripgrep

도구는 셋이다. 파이썬의 설치와 실행을 관리하는 uv, 실습을 싣는 Python 3.14, 그리고 7장에서 파일 트리 검색에 쓸 ripgrep 15. 정규식 모듈 `re`는 설치 목록에 없다. Python 표준 라이브러리에 들어 있는, 파이썬 안의 실습 무대다. 설치 채널은 둘로 정리된다. uv와 ripgrep은 OS 패키지 관리자로 설치한다 — macOS는 Homebrew, Windows는 winget. 파이썬은 그 uv가 설치하고 실행하므로, python.org의 설치 파일은 이 코스에서 쓰지 않는다. 담로의 명부는 작업대가 선 다음에 편다. 이 절이 그 작업대다.

설치는 순서가 절반이다. 선행 조건부터 본다.

**macOS** — Homebrew가 선행 설치물이다. `brew` 명령이 없다면 brew.sh의 설치 명령 한 줄을 실행한다. 설치기는 Command Line Tools(`xcode-select --install`) 요구까지 알아서 처리하고, 끝나면 "Next steps" 아래에 셸 프로필용 명령 두 줄을 출력한다. 그 두 줄을 그대로 복사해 실행하면 `~/.zshrc`에 `eval "$(/opt/homebrew/bin/brew shellenv)"`(Apple Silicon 기준 경로)가 들어간다. 새 터미널을 열어 `brew --version`이 숫자를 찍으면 선행 조건 통과다.

```
brew install uv
brew install ripgrep
uv --version            # 0.11.29 이상인지 숫자를 읽는다
uv python install 3.14
```

**Windows 11** — winget은 시스템 구성요소로 기본 제공되지만, 깨끗한 머신에서는 첫 로그인 직후 Microsoft Store의 비동기 등록이 끝나기 전까지 인식되지 않을 수 있다. 그래서 `winget --version` 확인이 먼저다.

```
winget --version
winget install --id astral-sh.uv -e
winget install --id BurntSushi.ripgrep.MSVC -e
```

첫 `winget install`에서는 소스 약관 동의를 묻는 프롬프트가 한 번 뜬다. Y를 입력하면 진행되고 다음부터는 묻지 않는다. 그리고 winget의 uv 설치는 portable 형식이라 PATH 반영에 새 세션이 필요하다. 터미널을 완전히 닫았다가 새로 열고 이어간다.

```
uv --version            # 0.11.29 이상인지 숫자를 읽는다
uv python install 3.14
```

설치가 끝났다는 판정은 자기 느낌이 아니라 검증 명령 세 개가 내린다. 양 OS 동일하다.

```
uv --version
uv run --python 3.14 python --version
rg --version
```

읽는 지점도 정해져 있다. uv는 0.11.29 이상, Python은 3.14.x, rg는 출력 첫 줄이 15.x 라인. 종료 코드가 0이라는 사실만으로는 부족하다 — 숫자를 눈으로 읽는다.

실습 셸은 OS 기본을 그대로 쓴다 — macOS는 zsh, Windows는 Windows Terminal이 띄우는 Windows PowerShell. 두 셸은 인용 규칙이 달라 7장에서 서로 어긋나는 지점이 오지만, 이 장의 명령은 양쪽에서 글자 하나까지 같다.

파이썬 대화형 셸(REPL) 진입 명령은 하나다.

```
uv run --python 3.14 python -X utf8
```

프롬프트가 `>>>`로 바뀌면 파이썬이 입력을 기다리는 상태다. 한 줄 넣으면 결과가 바로 돌아온다.

```text
>>> 1 + 1
2
>>> exit()
```

나가는 방법은 `exit()` — macOS는 Ctrl-D, Windows는 Ctrl-Z 후 Enter로도 나갈 수 있다.

진입 명령 끝의 `-X utf8` 한 조각이 이 코스의 규약이다. UTF-8 실행 규약(UTF-8 convention) — 모든 실습을 `-X utf8`(또는 환경 변수 `PYTHONUTF8=1`)로 실행하고, 실습용 파일은 UTF-8로 저장한다. 환경 변수 쪽을 쓴다면 zsh는 `export PYTHONUTF8=1`, PowerShell은 `$env:PYTHONUTF8="1"`이다. 근거는 OS 간 차이에 있다. Python 3.14는 UTF-8 모드가 아직 기본이 아니고, 한국어 Windows의 로캘 기본 인코딩은 cp949다. 규약 없이 파일을 읽고 쓰는 순간 같은 코드가 양 OS에서 다르게 움직인다. 규약 한 줄이 그 어긋남을 막는다. 조항이 하나 더 있다. ripgrep의 비ASCII 검색에는 `-E`로 인코딩을 명시한다는 것인데, 실제로 쓰는 곳은 7장이다.

지금 실행 환경이 규약대로인지 코드로 판정한다.
```python
import sys

print(sys.version_info.major, sys.version_info.minor)
print(sys.flags.utf8_mode)
```

실행 결과:

```text
3 14
1
```

**핵심 —** `utf8_mode`가 1이 아니라면 `-X utf8` 없이 진입한 것이다. 세션을 나가서 진입 명령부터 다시 실행한다.

이 설치 절차의 어디에 함정이 숨어 있는가. 공통에 둘, OS별로 하나씩이다. 공통부터 본다.

- `uv python install 3.14`를 건너뛰고 바로 `uv run --python 3.14 ...`를 실행하면 uv가 관리하는 파이썬 배포(managed distribution)를 조용히 내려받는다(수십 MB). 오동작은 아니지만 오프라인이면 실패하고, 첫 실행이 이유 없이 느려 보인다. 설치 단계를 명시적으로 밟는 이유다.
- 버전 확인을 종료 코드만 보고 넘기면 낡은 설치를 놓친다. 기준 숫자는 앞의 검증 명령 세 개 옆에 적어 두었다.

**Windows** — `python`을 그대로 부르는 일이 없다. 이 코스의 명령은 전부 uv를 거치고, uv가 관리 배포를 PATH보다 우선 해석한다. 앱 실행 별칭 스텁이 Microsoft Store 창을 여는 고전적 함정도 그래서 이 코스가 검증하는 명령 집합에는 애초에 들어올 자리가 없다. 습관적으로 `python`을 직접 입력했다면 이야기가 다르다. Store 창이 뜨면 안내를 따라가 Store판 파이썬을 설치하지 말 것(코스 채널 밖이다) — 대신 설정의 'Manage app execution aliases'에서 App Installer의 python.exe와 python3.exe 별칭을 끈다. 참고로 uv가 설치한 파이썬은 `py -V:Astral/CPython3.14`로도 실행된다.

**macOS** — `/usr/bin/python3`(시스템 파이썬 3.9.6)이 깔려 있어도 무시한다. 이 코스의 명령이 부르지 않으니 거짓 통과 위험은 없지만, 직접 `python3`를 입력하면 Command Line Tools 설치 팝업이 뜨거나 낡은 버전이 응답한다. Command Line Tools 요구는 Homebrew 선행 설치 단계에서 이미 끝난 문제다.

다음 세션에 돌아와 이 장을 이어갈 때의 절차. 설치를 다시 할 필요는 없다.

1. 터미널을 열고 `uv run --python 3.14 python --version` 한 줄로 환경 생존을 확인한다. 3.14.x가 나오면 그대로 진행.
2. `uv run --python 3.14 python -X utf8`로 REPL에 다시 진입한다.
3. REPL의 변수는 세션과 함께 사라진다. 지난 세션에서 정의한 명부 문자열은 이미 없다 — 진도가 1.2절까지 갔다면, 그 절 첫 예제의 명부 정의부터 다시 붙여 넣고 시작하면 된다.

**확인.** 모든 실습을 `-X utf8`(또는 `PYTHONUTF8=1`)로 실행하는 이유는 무엇인가?

Python 3.14는 UTF-8 모드가 기본이 아니고 한국어 Windows의 로캘 기본 인코딩은 cp949라서, 파일을 읽고 쓰는 코드가 OS에 따라 다르게 동작한다. 규약으로 실행 모드를 고정해 양 OS에서 같은 결과를 보장한다.

*책을 덮고 — 설치한 도구 셋과 각각의 검증 명령, 그리고 각 출력에서 읽어야 할 숫자를 떠올려 보라.*

## 1.2 첫 매칭 — 명부에서 배 찾기

담로가 이번 주 입출항 기록 넉 줄을 내민다. 이 넉 줄이 장 끝까지 쓸 재료다.

```text
은파호 입항 4721-05-03
달무리호 출항 4721-05-03
바람개비호 입항 4721-05-04
첫서리호 점검 4721-05-05
```

이 명부에서 "은파호"를 찾는 가장 단순한 정규식은 `은파호` 그 자체다. 메타문자 없이 문자가 자기 자신과 매칭되는 패턴을 리터럴 패턴(literal pattern)이라 부른다. 이 작은 언어의 가장 단순한 문장에 해당한다. 뜻은 하나 — "이 글자들이 이 순서로 붙어 나오는 자리를 찾아라". 이 정도라면 `str.find`로도 되는 일이다. 그런데도 언어라 부르는 이유는 패턴 자리에 문자열 하나가 아니라 "문자열 집합의 서술"이 들어갈 수 있어서다. 다음 장에서 대괄호 하나가 그 문을 연다.

찾는 함수는 `re.search`다. 문자열을 왼쪽부터 훑다가 패턴이 처음 성립하는 자리에서 멈추고, 그 자리의 정보를 담은 매치 객체(match object)를 돌려준다. 담로의 손가락도 같은 절차를 밟는다. 짚는 자리마다 찾는 이름 전체를 대보고, 맞아떨어진 자리에서 멈춘다.

이 장의 예제는 파일로 저장해 `uv run --python 3.14 python -X utf8 파일명.py`로 실행하는 스크립트 형태다. REPL에 줄 단위로 붙여 넣어도 결과는 같다.

같은 명부에 검색 두 번 — 한 번만 나오는 이름과 두 번 나오는 낱말을 각각 던진다.
```python
import re

manifest = """은파호 입항 4721-05-03
달무리호 출항 4721-05-03
바람개비호 입항 4721-05-04
첫서리호 점검 4721-05-05"""

print(re.search("은파호", manifest))
print(re.search("입항", manifest))
```

실행 결과:

```text
<re.Match object; span=(0, 3), match='은파호'>
<re.Match object; span=(4, 6), match='입항'>
```

**핵심 —** "입항"은 명부에 두 번 있지만 보고는 한 건뿐이다. `re.search`는 가장 왼쪽의 첫 매치에서 멈춘다. 전부 찾기가 아니라 첫 발견 보고다.

출력의 `span=(0, 3)`이 판정의 알맹이다. 매치가 인덱스 0에서 시작해 3 직전에서 끝났다는 뜻으로, 파이썬 슬라이스와 같은 반개구간으로 읽으면 된다. 둘째 검색의 `span=(4, 6)`은 판정 절차의 흔적까지 보여준다. 엔진은 인덱스 0에 손가락을 놓고 "입항" 두 글자 전체를 대본다 — 첫 글자부터 어긋나니 실패. 인덱스 1, 2, 3에서도 똑같이 실패한다. 인덱스 4에서 비로소 '입'과 '항'이 연달아 맞아떨어지고, 엔진은 거기서 멈춰 (4, 6)을 기록한다. 그리고 매치 객체는 화면에 찍힌 문자열이 아니라 물어볼 수 있는 객체다. 매치된 문자열이 무엇인지, 어디서 시작해 어디서 끝났는지를 메서드로 묻는다.

매치 객체에서 매치 문자열과 위치를 각각 꺼낸다.
```python
import re

manifest = """은파호 입항 4721-05-03
달무리호 출항 4721-05-03
바람개비호 입항 4721-05-04
첫서리호 점검 4721-05-05"""

m = re.search("바람개비호", manifest)
print(m.group())
print(m.span())
print(m.start(), m.end())
```

실행 결과:

```text
바람개비호
(37, 42)
37 42
```

**핵심 —** 시작이 왜 37인가. 개행도 한 글자여서, 앞의 두 줄과 개행 문자 두 개가 모두 인덱스에 세어진다.

**확인.** 명부에 "입항"이 두 번 있는데 `re.search("입항", manifest)`는 왜 한 건만 보고하는가?

<details>
<summary>답 확인</summary>

`re.search`의 계약이 "가장 왼쪽 첫 매치 하나"이기 때문이다. 왼쪽부터 훑다가 첫 성공 지점에서 멈춘다 — 모든 매치를 모으는 일은 다른 함수에 넘어가 있다.

</details>

**확인.** 매치의 시작과 끝 인덱스를 한 번에 튜플로 얻는 메서드는?

<details>
<summary>답 확인</summary>

`m.span()`. 반환은 (시작, 끝) 반개구간이고, `m.start()`·`m.end()`와 같은 값이다.

</details>

*방금 절을 책을 덮고 재구성해 보라 — `re.search`는 어디서 시작해 언제 멈추고, 성공의 증거로 무엇을 돌려주는가.*

## 1.3 못 찾았다는 대답, None

청람항의 배는 넷만 있다. 유령호라는 배는 명부 어디에도 없다 — 없는 배를 가리킬 때 항구 사람들이 쓰는 이름이다. 없는 것을 찾으면 무엇이 돌아오는가. 판정의 나머지 절반이 여기 있다.

명부에 없는 이름을 검색한 결과를 그대로 출력한다.
```python
import re

manifest = """은파호 입항 4721-05-03
달무리호 출항 4721-05-03
바람개비호 입항 4721-05-04
첫서리호 점검 4721-05-05"""

print(re.search("유령호", manifest))
```

실행 결과:

```text
None
```

**핵심 —** 빈 문자열도, False도, 오류도 아니다. "매치 없음"의 정확한 표현은 None. 돌려줄 매치 객체가 아예 없다는 뜻이다.

**예측.** 직전 예제를 실행한 REPL에서 이어서, 방금 그 실패한 검색의 반환값에 바로 `.group()`을 부른다.

```python
m = re.search("유령호", manifest)
print(m.group())
```
빈 문자열이 출력될까, 아니면 다른 일이 벌어질까? 실행 전에 답을 적어 보라.

<details>
<summary>답 확인</summary>

`AttributeError: 'NoneType' object has no attribute 'group'` — 프로그램이 오류로 죽는다. `m`이 매치 객체가 아니라 None이기 때문이다.

</details>

<details>
<summary>올바른 모델</summary>

많은 검색 API가 실패를 빈 값(빈 문자열, 빈 리스트)으로 돌려주기에 "실패해도 `.group()`은 빈 결과겠지"라는 직관이 생긴다. `re.search`의 모델은 다르다 — 성공이면 매치 객체, 실패면 None이라는 두 갈래 반환이고, None에는 `.group()`이라는 질문 자체를 던질 수 없다. 실패를 빈 문자열로 표현하지 않는 데도 이유가 있다. 성공이 "빈 문자열과의 매치"일 수도 있어서다(3장에서 만난다). 실패까지 빈 값으로 표현하면 두 경우가 뒤섞인다. 그래서 관용구가 정해져 있다. 반환값을 변수로 받아 `if m:`으로 분기한 뒤에만 그룹에 접근한다.

</details>

**주의 —** 실무 트레이스백에서는 이 오류가 수십 줄 아래 파묻혀 나온다. 알아보는 표식은 마지막 줄의 `NoneType`이다. 속성 이름이 `group`이 아니라 `start`나 `span`이어도 진단은 같다 — 검색이 실패했는데 코드는 성공을 전제했다. 패턴을 의심하기 전에 반환값 분기부터 넣는다.

이 분기 관용구가 None 가드(None guard)다. 매치 객체는 truthy, None은 falsy라서 `if m:` 한 줄이 두 세계를 가른다.

있는 배와 없는 배를 같은 코드로 처리한다 — 가드가 두 경우를 가른다.
```python
import re

manifest = """은파호 입항 4721-05-03
달무리호 출항 4721-05-03
바람개비호 입항 4721-05-04
첫서리호 점검 4721-05-05"""

for ship in ["첫서리호", "유령호"]:
    m = re.search(ship, manifest)
    if m:
        print(f"{ship}: 기록 있음 (위치 {m.start()})")
    else:
        print(f"{ship}: 기록 없음")
```

실행 결과:

```text
첫서리호: 기록 있음 (위치 57)
유령호: 기록 없음
```

**핵심 —** `if m is not None:`을 길게 쓰지 않는 것이 관례다. 매치 객체의 truthiness가 그 축약을 보증한다.

**확인.** `re.search`가 실패했을 때의 반환값은 무엇이고, 그 값에 `.group()`을 부르면 어떻게 되는가?

<details>
<summary>답 확인</summary>

반환값은 None이다. 거기에 `.group()`을 부르면 AttributeError로 죽는다 — `if m:` 분기가 선행돼야 한다.

</details>

*실패를 알리는 값과, 그 값 때문에 정해진 관용구를 코드 한 줄로 떠올려 보라.*

## 1.4 백슬래시의 두 층 — raw 문자열

판정 절차에는 사실 한 단계가 더 숨어 있다. 패턴이 정규식 엔진에 도달하기 전에 파이썬 문자열 리터럴이라는 층을 먼저 통과한다는 것. `"은파호"` 같은 순한 패턴에서는 이 층이 투명해서 존재조차 느껴지지 않는다. 백슬래시가 들어오는 순간 그 투명하던 층이 전면에 나선다.

정규식에는 백슬래시로 시작하는 표기가 많다. 4장에서 다룰 단어 경계 `\b`도 그중 하나다. 문제는 파이썬 문자열 리터럴에게도 `\b`가 자기 표기라는 점이다 — 백스페이스 문자(ASCII 8). 같은 두 글자를 두 언어가 서로 자기 것이라 주장한다. 패턴이 문자열 층과 정규식 층의 해석을 차례로 통과한다는 이 그림이 두 층 모델이다.

**예측.** 직전 예제를 실행한 REPL에서 이어서, 따옴표만으로 쓴 `\b` 패턴을 명부 첫 줄에 던진다.

```python
print(re.search("\b입항", "은파호 입항 4721-05-03"))
```
"입항" 앞은 공백이라 단어 경계가 성립하는 자리다. 매치가 나올까?

<details>
<summary>답 확인</summary>

None — 찾지 못한다. 엔진에 도달한 패턴이 `\b입항`이 아니기 때문이다.

</details>

<details>
<summary>올바른 모델</summary>

백슬래시를 파이썬 문자열 리터럴과 정규식 엔진이 각자 해석한다는 것이 두 층 모델이다. `"\b"`는 엔진에 닿기 전에 문자열 층에서 백스페이스 문자(ASCII 8) 하나로 변환된다 — 엔진이 받은 패턴은 "백스페이스 뒤에 입항"이고, 명부에 백스페이스 문자는 없다. 오류 한 줄 없이 일어나는 이 조용한 변환이 이른바 backslash plague다. 접두사 r을 붙이면 문자열 층의 해석이 꺼져 두 글자 `\b`가 그대로 엔진에 전달된다.

</details>

문자열 층의 해석을 끄는 표기가 raw 문자열(raw string)이다. 접두사 r 하나로 백슬래시가 변환 없이 정규식 엔진까지 도착한다. 두 방식은 길이부터 다르다.

같은 두 글자를 두 방식으로 쓴 결과를 길이와 repr로 비교한다.
```python
plain = "\b"
raw = r"\b"
print(len(plain), repr(plain))
print(len(raw), repr(raw))
```

실행 결과:

```text
1 '\x08'
2 '\\b'
```

**핵심 —** 따옴표 버전은 엔진에 닿기 전에 이미 한 글자(백스페이스)다. 정규식 엔진은 이 변환이 끝난 뒤의 값만 받는다.

예측 문제의 검색을 raw 문자열 버전과 나란히 실행한다.
```python
import re

line = "은파호 입항 4721-05-03"
print(re.search("\b입항", line))
print(re.search(r"\b입항", line))
```

실행 결과:

```text
None
<re.Match object; span=(4, 6), match='입항'>
```

**핵심 —** r 한 글자가 판정을 갈랐다. 결론은 습관이다 — 패턴 문자열은 백슬래시 유무와 무관하게 항상 `r"..."`로 쓴다. 그러면 두 층 문제를 매번 따질 필요가 없다.

**확인.** `r"\b입항"`의 접두사 r이 끄는 해석은 어느 층의 해석인가? 정규식 층의 `\b` 의미도 함께 사라지는가?

<details>
<summary>답 확인</summary>

파이썬 문자열 리터럴 층의 백슬래시 해석만 꺼진다. 정규식 층은 그대로라 `\b`는 엔진에서 단어 경계로 해석된다 — r은 전달을 보장할 뿐, 패턴의 의미를 바꾸지 않는다.

</details>

*패턴 문자열이 엔진에 도달하기까지 통과하는 두 층의 이름과, 각 층에서 `\b`가 무엇으로 해석되는지 말해 보라.*

**문제.** `find_ship(name)` 함수를 작성하라. 명부에서 name을 검색해, 있으면 `"name: 위치 N"` 형식의 문자열을, 없으면 `"name: 기록 없음"`을 반환한다(출력이 아니라 반환). 네 척과 유령호, 총 다섯 이름으로 호출해 결과를 출력하는 스크립트로 완성하고 규약대로 실행하라.

<details>
<summary>답 확인</summary>

```python
import re

manifest = """은파호 입항 4721-05-03
달무리호 출항 4721-05-03
바람개비호 입항 4721-05-04
첫서리호 점검 4721-05-05"""

def find_ship(name):
    m = re.search(name, manifest)
    if m:
        return f"{name}: 위치 {m.start()}"
    return f"{name}: 기록 없음"

for name in ["은파호", "달무리호", "바람개비호", "첫서리호", "유령호"]:
    print(find_ship(name))
```
핵심은 함수 안의 None 가드다. 반환 경로가 성공과 실패의 두 갈래임을 함수 차원에서 처리했으므로, 호출하는 쪽은 분기 걱정 없이 결과 문자열만 쓴다.

</details>

**문제.** 동료 서기가 `re.search("\b점검", manifest)`로 점검 기록을 찾다가 None만 받고 "명부에 점검 기록이 없다"고 결론 내렸다. 두 층 모델로 이 진단이 왜 틀렸는지 설명하고, 코드를 두 가지 방법으로 고쳐라.

<details>
<summary>답 확인</summary>

패턴이 엔진에 도달하기 전에 문자열 층이 `\b`를 백스페이스 문자(ASCII 8)로 바꿨다. 엔진이 받은 패턴은 "백스페이스 뒤에 점검"이고 명부에 백스페이스 문자는 없으니, None은 올바른 판정이다 — 명부에 점검이 없는 것이 아니라 질문이 잘못 전달됐다. 교정 하나: `r"\b점검"`으로 raw 문자열을 써서 단어 경계가 엔진에 그대로 닿게 한다. 교정 둘: 이 명부에서는 리터럴 `"점검"`만으로 충분하다 — 위치 62에서 매칭된다.

</details>

**요약**

- 정규식은 문자열에 "찾았다/못 찾았다" 판정을 내리는 작은 언어이고, 리터럴 패턴은 그 언어의 가장 단순한 문장이다.
- `re.search`는 왼쪽부터 훑다가 가장 왼쪽 첫 매치에서 멈춘다. 성공의 증거가 매치 객체 — `group()`으로 매치 문자열을, `span()`·`start()`·`end()`로 위치를 묻는다.
- 실패의 표현은 None이다. `if m:` 가드를 거친 뒤에만 그룹에 접근한다 — 건너뛰면 AttributeError로 죽는다.
- 패턴은 파이썬 문자열 리터럴 층과 정규식 층을 차례로 통과한다. `\b`처럼 두 층의 해석이 갈라지는 표기가 있으므로 패턴은 항상 `r"..."`로 쓴다.
- 실행 규약 — 모든 실습은 `-X utf8`(또는 `PYTHONUTF8=1`)로 실행하고 실습 파일은 UTF-8로 저장. 양 OS 동일 결과의 전제다.

**다음 장에서는**

리터럴 패턴은 글자를 그 글자로만 찾는다. 다음 장의 질문은 "아무 문자"와 "이 문자들 중 하나" — 대괄호 한 쌍(문자 클래스)과 점 하나가 리터럴로는 쓸 수 없는 패턴을 열고, `\d` 같은 축약 클래스가 유니코드라는 복병과 함께 등장한다. 재료는 시장 장부다. 외래 상단의 표기 습관이 남긴 낯선 숫자들이 `\d`의 진짜 범위를 시험한다.
